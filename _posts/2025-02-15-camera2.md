## 1. 카메라 직접 움직이기 (키보드 조작)

루프를 도는 자동 카메라 워킹을 넘어, 직접 키보드 WASD 키를 눌러 1인칭 시점으로 씬을 탐험하는 컨트롤 시스템을 구축해 보자. 위치 연산을 위해 카메라의 기준점 변수 3가지를 전역 공간에 할당한다.

```cpp
glm::vec3 cameraPos   = glm::vec3(0.0f, 0.0f,  3.0f);
glm::vec3 cameraFront = glm::vec3(0.0f, 0.0f, -1.0f);  // 카메라가 응시하는 정면 방향 벡터
glm::vec3 cameraUp    = glm::vec3(0.0f, 1.0f,  0.0f);
```
{: file="main.cpp" }

뷰 행렬을 생성하는 `lookAt` 함수는 이제 렌더 루프 매 프레임마다 변동되는 이 동적 변수들을 참조하도록 세팅한다.

```cpp
// 타겟 위치 = 현재 발조심(Position) + 쳐다보는 시선 방향(Front 방향 벡터)
view = glm::lookAt(cameraPos, cameraPos + cameraFront, cameraUp);
```
{: file="main.cpp" }

이전에 창 종료(ESC) 감지를 위해 짰던 `processInput` 함수를 재활용하여 WASD 키보드 입력 핸들러 로직을 심어준다.

```cpp
void processInput(GLFWwindow *window)
{
    // ... ESC 처리 로직 ...
    const float cameraSpeed = 0.05f; 
    
    // W, S 키: 전진 및 후진. 현재 위치에서 전방(Front) 벡터를 더하거나 뺀다.
    if (glfwGetKey(window, GLFW_KEY_W) == GLFW_PRESS)
        cameraPos += cameraSpeed * cameraFront;
    if (glfwGetKey(window, GLFW_KEY_S) == GLFW_PRESS)
        cameraPos -= cameraSpeed * cameraFront;
        
    // A, D 키: 좌우 게걸음(Strafe). 전방(Front)과 위쪽(Up) 벡터를 외적하여 가로축(Right) 방향을 구한 뒤 그 방향으로 이동시킨다.
    if (glfwGetKey(window, GLFW_KEY_A) == GLFW_PRESS)
        cameraPos -= glm::normalize(glm::cross(cameraFront, cameraUp)) * cameraSpeed;
    if (glfwGetKey(window, GLFW_KEY_D) == GLFW_PRESS)
        cameraPos += glm::normalize(glm::cross(cameraFront, cameraUp)) * cameraSpeed;
}
```
{: file="main.cpp" }

> [!important]
> 좌우 이동 키(A, D) 구현부에서 교차 외적(Cross) 산출 후 무조건 **`glm::normalize` 로 정규화**를 때려 벡터 길이를 1로 맞춘 점에 주목하라. 이를 누락하면 대각선 이동 시 전진/측면 벡터 길이가 무지성으로 합산되어 캐릭터가 비정상적으로 튀어 나가는(Run Boost) 버그가 터진다.

## 2. 프레임 동기화 이동 (Movement Speed & DeltaTime)

수치 변수 `cameraSpeed` 를 루프 1바퀴당 0.05 이동으로 하드코딩 해버리면 치명적인 결함이 발생한다.
구형 사무용 PC(초당 30루프)에서는 캐릭터가 굼벵이처럼 기어가고, 하이엔드 게이밍 PC(초당 300루프)에서는 1초만에 맵 밖으로 순간 이동할 것이다. 컴퓨터 성능에 구애받지 않고 **"1초당 동일한 거리"를 안정적으로 걷게 하려면, 프레임 처리 지연 시간인 `deltaTime` 을 이동 계수로 곱해줘야 한다**.

`deltaTime` 이란 직전 프레임을 렌더링하고 모니터 스크린에 쏘기까지 걸린 현실의 시간 간격(초)이다.

```cpp
// 프레임 딜레이 타이머를 전역에 선언
float deltaTime = 0.0f;  
float lastFrame = 0.0f;  
```
{: file="main.cpp" }

루프의 정점 부분에서 매 프레임 시간 격차를 최신화한다.
```cpp
float currentFrame = glfwGetTime();
deltaTime = currentFrame - lastFrame;
lastFrame = currentFrame;
```
{: file="main.cpp" }

이제 하드웨어 클럭 깡패들에게 의존하지 않는 규격화된 이동 속도를 `processInput` 내부에서 산출한다.

```cpp
void processInput(GLFWwindow *window)
{
    // 속도 상수 2.5 단위를 세팅하고 딜레이 타임을 팩터로 묶어 곱셈
    float cameraSpeed = 2.5f * deltaTime;
    // ... [하단 프론트/사이드 이동 로직은 동일]
}
```
{: file="main.cpp" }

## 3. Look Around (마우스 시점 뷰 제어)

걸어 다니는 법을 배웠으니, 마우스 커서를 빙글빙글 돌려 고개를 꺾는 1인칭 프리캠 시스템을 연동할 차례다.

![Pitch Yaw Roll](https://learnopengl.com/img/getting-started/camera_smooth.png){: width="500" }  
_비행 시뮬레이터에서 유래된 Euler Angles 축_

### 3.1. 오일러 각도 (Euler Angle) 
오일러 각도는 3D 공간의 회전을 3축으로 쪼개어 정의한다. 바로 **Pitch(상하 끄덕임), Yaw(좌우 도리도리), Roll(좌우 갸우뚱)** 이다. 
이 3개의 축 각도를 정밀하게 세팅하고 융합하면 어떤 기괴한 시점 회전 벡터라도 계산이 가능하다. 일반적인 FPS 카메라에서는 플레이어의 고개가 꺾이는 멀미를 방지하기 위해 Roll 회전을 원천 봉쇄하고 xy 평면 느낌의 `pitch` 와 `yaw` 값만 추적 관리한다.

### 3.2. 삼각함수로 방향 벡터 유추하기

![Triangle](https://learnopengl.com/img/getting-started/camera_triangle.png){: width="400" }  

밑변이 1로 확정된 직각삼각형에서의 삼각함수 공식은 매우 직관적이다.
* `인접변 (밑변) = cos(각도)`
* `대향변 (높이) = sin(각도)`

이 공식을 마우스 시점 좌표계에 밀어 넣어, Z/X 평면 상에서 좌우 **Yaw 각도** 회전 시 축의 길이가 어떻게 증감하는지 투영해보자.

![Yaw Vector](https://learnopengl.com/img/getting-started/camera_yaw.png){: width="400" }    

만약 카메라의 렌즈 방향(Y축)을 기준으로 X축 평면 방향을 치켜보고(각도를 발생시키고) 삼각함수를 대입한다면 시점 방향 벡터의 파편을 아래와 같이 조립할 수 있다.

```cpp
glm::vec3 direction;
// C++의 sin, cos 함수 입력 인자는 반드시 '라디안(Radian)' 체계여야 한다.
direction.x = cos(glm::radians(yaw)); 
direction.z = sin(glm::radians(yaw));
```
{: file="main.cpp" }

이제 X축과 Z축 투영 길이를 도출했으니 하늘과 땅을 보는 **Pitch 각도**까지 구면 변환 방정식에 결합시켜 입체 좌표를 총망라해 준다.

```cpp
// 3축 오일러 시야각 최종 계산식
direction.x = cos(glm::radians(yaw)) * cos(glm::radians(pitch));
direction.y = sin(glm::radians(pitch)); // 하늘/땅 Y축은 오직 Pitch 담당이다.
direction.z = sin(glm::radians(yaw)) * cos(glm::radians(pitch));
```
{: file="main.cpp" }

## 4. 카메라 초기 시점 방향 불일치(Mismatch) 이슈

위의 완벽해 보이는 삼각 함수 수학 공식에는 숨겨진 치명적 함정이 있다. 

초기 환경에서 `yaw = 0`, `pitch = 0` 인 정자세 상태로 앱이 켜진다고 가정해보자. (사실 `pitch` 값이 뭐든 `yaw = 0`의 결과는 파멸적이다).

### 4.1. Yaw = 0° 수식 붕괴 과정
```text
direction.x = cos(0) * cos(0) => 1 * 1 = 1
direction.y = sin(0)          => 0
direction.z = sin(0) * cos(0) => 0 * 1 = 0
```
결과적으로 뽑혀 나오는 Direction 전방 벡터는 `(1, 0, 0)` 이다. 
이는 플레이어 카메라가 시작 모니터 정면 중앙 깊숙한 곳(Z축 음의 방향)을 쳐다보지 않고, **생뚱맞게 화면의 우측 90도 모서리 방향(X축 양의 방향)을 쳐다보며 시작한다**는 어처구니없는 뜻이다. 분명 씬 내의 렌더링 오브젝트들은 Z축의 음의 좌표 방향 쪽에 모아 뒀는데 말이다.

### 4.2. 해결책: 초기 Yaw 각도 오프셋 보정

카메라 렌즈가 기본적으로 사용자가 인지하는 모니터 정면 깊숙한 곳(음의 Z축 역방향)을 향해 조준되도록 하려면 꼼수가 필요하다. 
현재 `(1, 0, 0)` X축 양의 방향을 쳐다보고 있는 렌즈 고개를 정면 Z축이 있는 곳으로 **시계 방향 90도 회전** 꺾어버리면 직빵이다.

OpenGL 수학 체계에서는 **양수 각도가 반시계(CCW) 회전을 의미**하므로, 시계 방향으로 꺾어 내리려면 초기 `yaw` 변수 값 자체를 0이 아니라 **`-90.0f`** 로 억지로 시작하게 세팅하면 된다.

```cpp
// 전역 마우스 좌표계의 초기 편향 각도 부여
float yaw = -90.0f; 
```
{: file="main.cpp" }

이렇게 `-90`도 초기 값을 맥여 준 채 렌더 루프를 돌리고 계산식을 태우면 수식의 도출 벡터가 비로소 `(0, 0, -1)` 이 떨어지며 완벽하게 음의 Z축 평원을 쳐다보게 되며 모순이 해결된다.
