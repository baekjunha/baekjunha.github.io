## 1. 윈도우 생성 (Hello Window) 관련 Q&A

### 1.1. 왜 `glad`를 항상 `glfw`보다 먼저 `#include` 해야 하는가?

**결론부터 말하자면, 헤더 포함 순서에 따라 OpenGL 함수 선언과 실제 드라이버 함수 포인터 간의 충돌이 발생하기 때문입니다.**

1.  **GLFW의 내부 동작**: `GLFW/glfw3.h`는 디스플레이 환경과 OpenGL을 연동하기 위해 내부적으로 시스템 기본 OpenGL 헤더(`GL/gl.h` 등)를 포함합니다. 만약 `glfw`를 먼저 포함하면 시스템 기본 헤더가 먼저 로드됩니다. 이 헤더는 최신 OpenGL 명세를 포함하지 않거나 함수 선언 수준에 머물러 있어 실제 구현 주소를 연결하지 못할 수 있습니다.
2.  **GLAD의 역할**: `glad`는 이러한 함수 선언부에 현재 시스템의 그래픽 드라이버가 제공하는 실제 함수 메모리 주소(Pointer)를 동적으로 검색하여 할당합니다. 따라서 `glad`가 먼저 로드되어 런타임 함수 맵핑 테이블을 완전히 구축해야만, 이후 `glfw`가 해당 함수들을 정상적으로 호출하여 사용할 수 있습니다.
3.  **기술적 결과**: 순서를 어겨 `glfw`가 먼저 로드되면 `glad`의 포인터 할당 로직이 정상 동작하지 않을 수 있습니다. 이 경우 `glClear`나 `glViewport`와 같은 API 호출 시 런타임 에러 또는 `undefined reference`와 같은 링킹 에러가 발생하며 OpenGL API를 정상적으로 호출할 수 없게 됩니다.

> [!warning]
> **권장 사항**
> C/C++ 모던 OpenGL 환경에서는 `#include <glad/glad.h>` 구문이 `#include <GLFW/glfw3.h>`보다 선행하도록 배치하는 것이 원칙입니다.

### 1.2. OpenGL Context란 정확히 무엇인가?

**Context는 OpenGL의 전역 상태(Global State)와 모든 렌더링 동작을 관리하는 가상 작업 환경입니다.** 

OpenGL은 **상태 머신(State-machine) 기반 API**이므로, 모든 그래픽 명령(예: 색상 지정, 텍스처 바인딩)은 현재 해당 스레드에서 **활성화(Active)** 된 Context 내부의 상태값에 영향을 미칩니다. 

Context가 관리하는 주요 정보는 다음과 같습니다.
*   **렌더링 대상 상태**: 뷰포트(Viewport) 설정, 컬러 버퍼, 깊이(Depth) 및 스텐실(Stencil) 버퍼 정보.
*   **GPU 리소스 레퍼런스**: 컴파일된 셰이더 프로그램, 텍스처 객체, VBO/VAO 등의 객체 식별자.
*   **전역 렌더링 옵션**: 블렌딩(Blending), 깊이 테스트, 컬 페이스(Cull face) 등의 활성화 여부 및 매개변수.

`glfwCreateWindow` 함수를 호출하면 OS 윈도우 생성과 동시에 해당 윈도우에 대응하는 Context가 생성됩니다. 그러나 생성 직후에는 해당 Context가 현재 실행 중인 스레드에 연결되어 있지 않습니다. 반드시 **`glfwMakeContextCurrent(window)`**를 호출하여 해당 Context를 현재 스레드의 활성 작업 환경으로 지정(Binding)해야 합니다. 이 과정을 누락하면 이후 호출되는 모든 OpenGL 명령(`gl*` 함수)은 활성화된 Context를 찾지 못해 무시되거나 에러를 유발합니다.

### 1.3. `glfwMakeContextCurrent` 와 `glfwSetFramebufferSizeCallback`의 적절한 호출 시점

```cpp
glfwMakeContextCurrent(window);
glfwSetFramebufferSizeCallback(window, framebuffer_size_callback);
```
{: file="main.cpp" }

이 함수들은 윈도우 객체가 성공적으로 생성된 직후, 본격적인 리소스 초기화나 렌더 루프 진입 전에 최우선으로 호출되어야 합니다.

1.  **`glfwMakeContextCurrent(window)`**: Context를 활성화하는 단계입니다. **GLAD 초기화나 셰이더 객체 생성 등 GPU 리소스를 제어하는 모든 작업은 반드시 활성화된 Context가 존재할 때만 유효합니다.** 따라서 초기화 과정의 가장 앞부분에 위치해야 합니다.
2.  **`glfwSetFramebufferSizeCallback(window, framebuffer_size_callback)`**: 윈도우 리사이징 이벤트 발생 시 실행될 콜백 함수를 등록하는 과정입니다. 윈도우 크기 변경 시 뷰포트를 재설정하는 로직이 안전하게 동작하려면, Context가 안정적으로 활성화된 상태에서 등록하는 것이 바람직합니다.

### 1.4. 메인 루프 내 processInput과 glfwPollEvents의 역할 및 상호작용

`while(!glfwWindowShouldClose)`로 정의된 렌더 루프 내에서, 두 함수는 입력 장치 처리를 분담합니다.

*   `**glfwPollEvents()**`: 운영체제로부터 전달되는 키보드, 마우스, 윈도우 이벤트 등을 수집하여 처리합니다. GLFW는 내부적인 **이벤트 큐(Event Queue)**를 통해 이를 관리하며, 매 프레임 이 함수가 호출될 때 큐의 내용을 갱신하고 등록된 콜백 함수를 실행합니다.
*   `**processInput(window)**`: 수집된 이벤트를 바탕으로 응용 프로그램 수준에서 구체적인 반응을 정의합니다. 예를 들어 `glfwGetKey`를 통해 특정 키의 입력 상태를 확인하고 카메라의 위치를 수정하는 등의 로직을 수행합니다.

**상호작용 원리**
`glfwPollEvents()`가 선행되어 OS 수준의 입력 상태를 갱신해야만, `processInput`에서 현재 프레임의 정확한 입력 상태를 조회할 수 있습니다. 즉, `glfwPollEvents()`를 통해 최신화된 상태값이 `processInput`의 판단 근거가 되는 구조입니다.

## 2. 삼각형 그리기 (Hello Triangle) 관련 Q&A

### 2.1. 셰이더 소스 코드를 C++ 문자열로 정의하는 이유

셰이더는 C++ 런타임이 아닌 **GPU의 병렬 연산 장치에서 실행되는 독립적인 프로그램**입니다. 이는 **GLSL(OpenGL Shading Language)**로 작성되며, OpenGL API는 이 소스 코드를 텍스트 문자열 형식으로 전달받아 런타임에 컴파일합니다.

```glsl
const char *vertexShaderSource = "#version 330 core\n"
    "layout (location = 0) in vec3 aPos;\n"
    "void main()\n"
    "{\n"
    "   gl_Position = vec4(aPos, 1.0);\n"
    "}\0";
```
{: file="main.cpp" }

주요 구문 분석:
*   `**#version 330 core**`: 사용할 GLSL 버전과 코어 프로파일 스펙을 명시합니다.
*   `**layout (location = 0) in vec3 aPos;**`: 특정 속성 위치(Attribute Location 0)로 유입되는 정점 데이터를 `aPos` 변수에 할당합니다.
*   `**gl_Position = vec4(aPos, 1.0);**`: 정점 셰이더의 최종 결과물인 클립 공간(Clip Space) 좌표를 출력 변수인 `gl_Position`에 대입합니다. 3차원 좌표를 4차원 **동차 좌표(Homogeneous Coordinate)**로 변환하기 위해 마지막 성분에 1.0을 추가합니다.

> [!tip] 
> 코드 관리의 효율성을 위해, 실제 프로젝트에서는 셰이더 코드를 별도의 파일(`.vert`, `.frag`)로 관리하고 C++의 파일 스트림(예: `ifstream`)을 사용하여 로드하는 방식을 주로 사용합니다.

### 2.2. VAO 교체 시 `glBindVertexArray(0)` 호출의 필요성

**결론적으로, 새로운 VAO를 바인딩(`glBindVertexArray(new_vao)`)하면 이전에 바인딩되어 있던 VAO는 자동으로 비활성화(Unbind)됩니다.**

OpenGL Context는 한 번에 하나의 VAO만을 활성 상태로 유지할 수 있습니다. 따라서 여러 객체를 그리는 루프 내에서는 다음과 같이 직접 교체하는 것이 효율적입니다.

```cpp
// 효율적인 바인딩 교체 방식
glBindVertexArray(vao1);
glDrawArrays(...);

glBindVertexArray(vao2); // vao1은 자동 해제되고 vao2가 활성화됨
glDrawArrays(...);
```
{: file="main.cpp" }

`glBindVertexArray(0)`을 사용하여 명시적으로 바인딩을 해제하는 행위는 주로 초기화 단계에서 특정 객체의 설정을 마친 뒤, 이후의 다른 설정 작업이 의도치 않게 현재 객체에 영향을 주는 것을 방지하기 위한 **방어적 프로그래밍(Defensive Programming)** 관점에서 사용됩니다.

### 2.3. VBO에서 셰이더로의 데이터 전송 아키텍처

정점 데이터가 GPU로 전달되어 렌더링에 사용되는 과정은 다음과 같은 단계를 거칩니다.

1.  **VRAM 전송 (`glBufferData`)**: 시스템 메모리(RAM)에 있는 정점 배열을 그래픽 카드의 전용 메모리(VRAM) 내 버퍼 객체(VBO)로 복사합니다. 이는 데이터 전송 병목을 줄이기 위한 필수 과정입니다.
2.  **레이아웃 속성 정의 (`glVertexAttribPointer`)**: VRAM에 저장된 바이너리 데이터를 셰이더의 입력 속성에 맞게 해석하는 규칙을 설정합니다. 데이터의 타입, 개수, 스트라이드(Stride), 오프셋(Offset) 정보 등을 정의하며, 이 구성 정보는 **VAO(Vertex Array Object)**에 기록되어 보존됩니다.
3.  **렌더링 실행**: `glDrawArrays`와 같은 명령이 실행되면, GPU는 현재 바인딩된 VAO의 설정을 참조하여 VBO에서 데이터를 인출하고 파이프라인을 가동합니다.
    * **정점 셰이더**: 좌표 변환 등을 수행하여 정점을 클립 공간으로 투영합니다.
    * **프래그먼트 셰이더**: 래스터라이제이션을 거친 각 프래그먼트의 최종 색상을 결정합니다.

## 3. 카메라 시스템 및 수학적 원리 Q&A

### 3.1. 오일러 각(Euler Angles) 기반 시선 벡터 산출 원리

카메라의 시선 방향(`direction`)을 구할 때 Yaw와 Pitch 각도를 삼각함수로 결합합니다. 특히 X축 성분 계산식은 다음과 같습니다.

`direction.x = cos(glm::radians(yaw)) * cos(glm::radians(pitch));`

여기서 `cos(pitch)`를 곱하는 이유는 **카메라의 상하 기울기(Pitch)가 커질수록 수평 평면(X-Z 평면)으로 투영되는 벡터의 길이가 줄어드는 기하학적 특성을 반영하기 위함입니다.**

*   `**cos(yaw)**`: Pitch가 0일 때, 수평 회전에 따른 X축 방향 성분의 크기를 결정합니다.
*   `**cos(pitch)**`: Pitch 각도가 변함에 따라 수평 성분이 축소되는 비율(Scale Factor)을 적용합니다. 
    *   Pitch가 90도(수직 상단 주시)가 되면 `cos(90)`은 0이 되어 수평 성분(X, Z)을 0으로 만듭니다.
    *   이러한 보정 과정을 통해 시선 벡터가 왜곡 없이 구면 좌표계 상에서 정확한 방향을 가리키게 됩니다.

### 3.2. 마우스 오프셋(Offset) 계산 시 Y축 부호 처리의 이유

마우스 이동 변위를 계산할 때 X축과 Y축의 가감 순서가 다르게 설정되는 경우가 많습니다.

```cpp
float xoffset = xpos - lastX;
float yoffset = lastY - ypos; // Y축의 감산 순서가 반전됨
```
{: file="main.cpp" }

이는 **윈도우 OS의 스크린 좌표계와 3D 공간 좌표계 간의 지향성 차이** 때문입니다.

1.  **스크린 좌표계**: 일반적인 윈도우 시스템에서 Y축은 상단에서 하단으로 갈수록 값이 증가합니다(Top-down).
2.  **조작 직관성**: 마우스를 위로 움직이면 사용자는 시야가 위로 향하기를 기대합니다. 하지만 윈도우 좌표계에서 위로 움직이면 `ypos` 값은 작아집니다.
3.  **부호 보정**: 만약 `ypos - lastY`를 그대로 사용하면 마우스를 위로 올릴 때 음수 값이 산출되어 시야가 아래로 내려가는 역전 현상이 발생합니다. 따라서 `lastY - ypos`로 순서를 바꾸어 마우스를 위로 올릴 때 양수(Pitch 증가)가 나오도록 논리적으로 보정하는 것입니다.
