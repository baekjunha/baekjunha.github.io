## 1. 윈도우 생성 (Hello Window) 관련 Q&A

### 1.1. 왜 `glad`를 항상 `glfw`보다 먼저 `#include` 해야 하는가?

**핵심 원인: 컴파일 단계에서의 OpenGL 함수 포인터 정의 충돌 방지**

C++ 코드 상단에서 `glad` 헤더를 `glfw`보다 먼저 포함하지 않으면, `glad`가 OpenGL 함수 포인터들을 정상적으로 훅킹(Hooking)하여 정의하지 못해 컴파일 에러가 터진다. 세부적인 이유는 다음과 같다.

1.  **GLFW의 내부 동작**: `GLFW/glfw3.h` 헤더 파일은 화면 입출력 장치와 OpenGL을 연동하기 위해 내부적으로 구형 OpenGL 헤더(`GL/gl.h` 등)를 자체적으로 끌어다 쓴다. 만약 코더가 `glfw`를 먼저 포함해버리면, 이 구형 기본 헤더가 먼저 컴파일러에 읽힌다. 이 구형 헤더는 최신 모던 OpenGL 함수들을 그저 스켈레톤(선언) 수준으로만 가지고 있을 뿐, 실제 메모리 상의 함수 구현 포인터를 연결하지 못한다.
2.  **GLAD의 역할**: `glad`의 존재 이유는 바로 이 비어있는 함수 뼈대에, 현재 컴퓨터에 장착된 그래픽 카드의 실제 드라이버 함수 메모리 주소(포인터)를 동적으로 찾아내 연결해 주는 것이다. 따라서 `glad`가 무조건 먼저 포함되어 올바른 최신 함수 포인터 맵핑 테이블을 싹 다 세팅해 두어야만, 그제서야 뒤따라 들어오는 `glfw`가 차려진 밥상(함수 포인터들)을 온전히 가져다 쓸 수 있다.
3.  **치명적 결과**: 만약 순서를 어겨 `glfw`가 먼저 읽히면, `glad`가 포인터 덮어쓰기를 실패할 확률이 높다. 이 상태에서 `glClear`나 `glViewport` 같은 그래픽스 함수를 호출하면, 컴파일러나 링커가 해당 함수의 실제 구현 주소를 찾지 못해 지옥의 `undefined reference` 링킹 에러를 무수한 경고 단위로 뿜어낸다.

> [!warning]
> **절대 규칙**
> 어떠한 상황에서도 C/C++ 모던 OpenGL 환경에서는 `#include <glad/glad.h>` 라인이 `#include <GLFW/glfw3.h>` 라인보다 무조건 윗줄에 위치해야 한다.

### 1.2. OpenGL Context란 정확히 무엇인가?

**Context는 OpenGL의 전역 상태(Global State)와 모든 렌더링 동작을 통제하는 하나의 거대한 '가상 작업 공간' 환경이다.** 

OpenGL은 전형적인 **거대 상태 머신(State-machine) 기반 API**이므로, 네가 내리는 모든 그래픽 명령(가령 "색을 빨간색으로 칠해라")은 오로지 현재 OS에 의해 **활성화(Active)** 된 Context 내부의 상태값만을 변경한다. 

Context가 품고 관리하는 방대한 정보는 다음과 같다.
*   **렌더링 캔버스 상태**: 뷰포트 크기, 컬러 버퍼 피트, 깊이(Depth) 버퍼, 스텐실(Stencil) 버퍼 등 스크린 정보를 담는다.
*   **연결된 GPU 리소스 내역**: 컴파일된 셰이더 프로그램, 텍스처 메모리 주소, VBO/VAO(Vertex Array Object) 등 모든 객체의 레퍼런스를 추적한다.
*   **렌더링 전역 옵션**: 블렌딩 규칙, 깊이 테스트 활성 여부, 컬 페이스(Cull face) 동작 등 글로벌 제어 변수들.

C++에서 `glfwCreateWindow` 함수를 호출하면, OS에 창(Window) 객체를 띄움과 동시에 이면에서 이 거대한 Context 환경 하나를 세트로 생성해 준다. 하지만 그저 '생성'만 해두었을 뿐, 작업자가 아직 삽을 들고 그 공간에 들어간 것은 아니다. 반드시 **`glfwMakeContextCurrent(window)`** 함수를 이어서 명시적으로 호출해야만, 비로소 해당 창의 Context가 스레드의 "현재 포커스 잡힌 작업 공간"으로 세팅된다. 이 호출 과정을 누락하면 네가 치는 모든 OpenGL `gl*` 명령 함수들은 허공에 삽질을 하는 꼴이 되어 전부 무시당한다.

### 1.3. `glfwMakeContextCurrent` 와 `glfwSetFramebufferSizeCallback`의 적절한 호출 시점

```cpp
glfwMakeContextCurrent(window);
glfwSetFramebufferSizeCallback(window, framebuffer_size_callback);
```
{: file="main.cpp" }

위 두 마스터 셋업 함수는 반드시 창이 성공적으로 생성된 직후, 본격적인 렌더링 로직이나 GLAD 초기화가 들어가기 직전에 최우선으로 호출되어야 한다.

1.  **`glfwMakeContextCurrent(window)`**: 앞서 설명했듯 캔버스를 펼치는 작업이다. **GLAD 함수의 로딩, 셰이더 객체의 생성 등 GPU를 건드리는 모든 OpenGL 초기화 작업은 Context가 현재 스레드에 꽉 물려(활성화되어) 있을 때만 동작한다.** 그러니 무조건 `window` 생성 즉시 이 함수부터 갈겨야 한다.
2.  **`glfwSetFramebufferSizeCallback(window, framebuffer_size_callback)`**: 사용자가 마우스로 창 테두리를 잡아끌어 크기를 리사이징할 때마다 실시간으로 반응할 이벤트 리스너(콜백 함수)를 등록하는 과정이다. 창 크기 변경 이벤트 처리는 GLFW가 관장하므로, `window` 객체와 연결된 Context가 온전히 활성화된 안정적인 상태에서 이 리스너를 달아주어야, 훗날 콜백 함수 내부에서 `glViewport` 로 화면 렌더링 좌표계를 재조정할 때 뻑(Crash)이 나지 않는다.

### 1.4. 무한 렌더 루프 속 `processInput`과 `glfwPollEvents`의 역할 분담

오브젝트가 살벌하게 돌아가는 `while(!glfwWindowShouldClose)` 렌더링 루프 코어 안에서, 이 두 녀석은 입력 장치 처리를 양분하여 환상적인 티키타카를 수행한다.

*   `**glfwPollEvents()**`: 윈도우 OS가 던져주는 키보드 타건, 마우스 XY 좌표 이동, 버튼 클릭, 창 최소화 등 잡다한 풀(Full) 이벤트를 한 호흡에 싹 다 수거하고 처리하는 진공청소기다. GLFW는 내부적으로 **이벤트 큐(Event Queue)** 를 파놓고 기다리는데, 매 프레임 이 함수가 지시를 내릴 때마다 큐에 차곡차곡 쌓인 시스템 이벤트들을 모조리 빼내어 OS에 보고를 넣고, 네가 앞서 등록해둔 잡다한 콜백 이벤트(마우스 클릭 등)들을 터뜨려 실행시킨다.
*   `**processInput(window)**`: 네가 C++ 영역에서 커스텀하게 짜놓은 구체적인 입력 반응 로직 함수다. 가령 `glfwGetKey` 로 W, A, S, D 키가 찰나의 순간에 눌려 눌린 상태인지를 체크하고, 눌려 있다면 플레이어 카메라의 포지션을 이동시키는 식의 '구체적인 인게임 조작'을 전담한다.

**필연적 인과관계**
OS단에서 키보드 불빛이나 마우스 센서 이동을 감지해 GLFW 내부 큐에 집어넣는 그 원천 작업의 트리거 자체가 `glfwPollEvents()` 다. 다시 말해, 네가 만든 `processInput` 함수가 WASD 키를 아무리 죽어라 읽으려 조회해 봤자, 그 앞에 `glfwPollEvents()` 님이 선행 포문을 열어 외부 이벤트를 갱신해 주지 않으면 평생 과거의 키보드 상태값만 들여다보는 환각에 빠지게 된다.

> **요약**: `glfwPollEvents()`가 "운영체제 센서 상태 업데이트!" 를 외치며 전체 키보드/마우스 판을 새로고침 갱신해 주면, `processInput` 이 "좋아, 지금 W키가 눌렸으니 캐릭터를 전진시킨다!" 라며 구체적 동작을 실행하는 메커니즘이다.

## 2. 삼각형 그리기 (Hello Triangle) 관련 Q&A

### 2.1. 굳이 셰이더 코드를 C++ 문자열(String) 변수에 우겨 넣는 이유는?

셰이더는 C++ 런타임 환경에서 도는 게 아니다. 이는 **온전히 GPU의 병렬 코어 유닛에서만 별개로 돌아가는 극도로 쪼그만 하드웨어 밀착형 프로그램**이며, 오로지 **GLSL(OpenGL Shading Language)**이라는 그래픽스 전용 C-Like 언어로만 작성된다.

우리가 C++ `.cpp` 파일 안에서 이걸 흉측한 하드코딩 문자열로 박아버리는 원초적 이유는, OpenGL API 라이브러리가 셰이더 소스 코드를 인가받는 인터페이스 창구가 **오로지 C-Style 텍스트 문자열(char pointer) 구문** 뿐이기 때문이다.

```glsl
const char *vertexShaderSource = "#version 330 core\n"
    "layout (location = 0) in vec3 aPos;\n"
    "void main()\n"
    "{\n"
    "   gl_Position = vec4(aPos, 1.0);\n"
    "}\0";
```
{: file="main.cpp" }

이 짧은 텍스트 파편을 뜯어보자.
*   `**#version 330 core**`: "난 모던 OpenGL 3.3 코어 스펙 GLSL 문법을 쓸 거야" 라고 GPU 런타임 컴파일러에게 선언하는 엄격한 헤더 명시다.
*   `**layout (location = 0) in vec3 aPos;**`: 파이프라인의 **0번째 슬롯(location=0)** 구멍으로 유입되는 데이터 스트림을 낚아채, `aPos` 라는 이름의 3차원(`vec3`) 벡터 변수에 담겠다고 입력(in) 인터페이스를 구축하는 거다.
*   `**gl_Position = vec4(aPos, 1.0);**`: 정점 셰이더의 핵심 임무. 입력받은 공간 x,y,z 3차원 좌표 끝에 스케일/투영 처리를 위한 가상의 W값 산수 역수 `1.0`을 달아 강제로 4차원 **동차 좌표(homogeneous coordinate)**로 빌드 업 시키고, 이를 시스템 기저 출력 변수인 `gl_Position`에 대입해버려 그래픽 카드의 래스터라이저 단계로 곧장 토스한다.
*   `**\0**`: C 계열 언어 포인터 열의 근본. "문장이 여기서 끝난다"는 널(null) 터미네이터 표식이다.

> [!tip] 
> 실무에서는 100줄, 1000줄 짜리 셰이더 코드를 C++ 내부에 하드코딩하면 가독성이 박살 뼈다귀가 되므로, 보통 `.vert`, `.frag` 확장자의 외부 순수 텍스트 파일로 코드를 분리 작성한 뒤, C++ 코드가 실행될 때 `ifstream` 같은 파일 스트림 모듈로 파일 내용을 몽땅 읽어와 `string` 버퍼에 던져 담는 방식을 표준으로 채택한다.

### 2.2. 무수한 VAO를 번갈아 쓸 때, 굳이 glBindVertexArray(0) 으로 언바인딩(초기화) 할 필요가 없는 논리

**결론부터 말하자면, 새로운 VAO 타겟을 바인딩(`glBindVertexArray(new_vao)`) 치는 즉시, 이전에 점거하고 있던 낡은 구형 VAO는 자동으로 결속이 튕겨져 나가 비활성화(Unbind)되기 때문이다.**

앞서 Context 개념에서 짚었듯 OpenGL 상태 머신의 연결 소켓은 단일 스레드 기반이므로 **한 번에 오직 단 하나의 VAO(Vertex Array Object) 결재판만 책상 위에 올려놓고(활성화) 결재 라인을 탈 수 있다.**

따라서 오브젝트 100개를 그리는 `while` 루프 안에서 아래처럼 짜는 건 코드 낭비다.

```cpp
// ❌ 하수들의 비효율적인 편집증 방식 (불필요한 CPU 콜백 낭비)
glBindVertexArray(vao1);
glDrawArrays(...);
glBindVertexArray(0); // vao1을 굳이 내려놓는다. (드랍 낭비)
glBindVertexArray(vao2);
glDrawArrays(...);

// ✅ 고수들의 플로우 타는 스왑(Swap) 방식
glBindVertexArray(vao1);
glDrawArrays(...);
glBindVertexArray(vao2); // vao1은 즉각 자동 퇴출되고 vao2가 제왕의 자리에 오른다.
glDrawArrays(...);
```
{: file="main.cpp" }

`glBindVertexArray(0)`을 던져서 명시적으로 소켓을 뽑아버리는 행동은, 초기화 파트에서 A 모델의 VAO 세팅 작업을 무사히 끝마친 직후 훗날 생성될 B 모델 VBO 데이터가 엉뚱하게 A 모델 VAO에 침류 되어 기록되는 대참사 사태를 방지하고자 방어적 코딩(Defensive Coding) 목적으로 보험 삼아 끊어주는 단발성 수비 플레이에 불과하다. 팍팍 돌아가는 라이브 렌더링 루프에서는 무지성 스왑(Swap)이 정답이다.

### 2.3. C++ 배열 데이터는 어떤 물리적 과정을 거쳐 셰이더의 혈관을 타는가?

단순한 CPU 램에 누워있던 멍청한 플로트 배열이 GPU의 셰이더 프로그램 연산 코어까지 살아서 흘러 들어가는 3단계 운송 프로세스는 가히 예술적이다.

1.  **택배 상차 (`glBufferData`)**: **메인보드 RAM(CPU 영역) 구석에 처박혀 있던 기가바이트 렌더링 정점 배열(`vertices`)을 PCIe 버스를 태워 그래픽스 카드 VRAM(GPU 메모리) 내부의 방대한 컨테이너 버퍼 화물칸(VBO)으로 냅다 덤프시켜 밀어 넣는다.** 이로써 병목의 주범인 CPU<->GPU 간의 왕복 통신 데이터 전송 작업은 끝났다.
2.  **화물칸 배선 라벨링 작업 (`glVertexAttribPointer`)**: **VRAM 화물칸(VBO)에 구겨 넣어진 1차원 이진수 부동소수점 데이터 더미들을, 어떤 스트라이드(간격)로 절삭 분해하여 셰이더의 몇 번 째 슬롯 파이프(Attribute) 변수 입구에 체결 시킬지 전선 라인을 이어주는 맵핑(Mapping) 통보 작업이다.** 
      * 예를 들어 `glVertexAttribPointer(0, 3, GL_FLOAT, ...)` 라는 건 "GPU야 지금부터 VBO 화물칸에서 데이터 뺄 때, float 단위 3개씩 묶어서 딱딱 잘라낸 다음, 셰이더 코드 안의 `layout (location = 0)` 이라고 라벨링된 `in vec3 aPos` 파이프 주둥이로 강제로 밀어 넣거라" 라고 결속력을 다지는 행위다. 그리고 이 배선 연결 기록은 모조리 **VAO** 가 녹화해 기억한다.
3.  **총 포문 발사 (셰이더 런타임 엔진 폭발)**: `glDrawArrays` 사격 명령이 떨어지는 순간, GPU는 VAO 결재판을 스캔하여 2번에서 맞춰둔 맵핑 파이프 밸브를 개방한다. VBO에서 데이터가 초당 수천만 번씩 불출되어 정점 셰이더 속으로 폭격을 가하고 들어간다.
      * **정점 셰이더(Vertex Shader)** 는 좌표계 변환 행렬을 곱해 `aPos` 포지션을 3D 공간상의 제 위치로 비틀고 뻗어 `gl_Position` 으로 투영 결산해 버린다.
      * **프래그먼트 셰이더(Fragment Shader)** 는 그렇게 조립된 철사 프레임 면적 내부 픽셀들을 낱낱이 도려내 찬란한 RGBA 픽셀 도색 시공 마감을 담당한다.

> **물류 데이터 직행 흐름**
> `CPU 마더보드 RAM 공간` → `[glBufferData 전송]` → `GPU VRAM 창고 (VBO)` → `[glVertexAttribPointer 배선]` → `VAO 기억 장치에 스냅샷 고정` → `[glDraw 사격]` → `GPU 내장 셰이더 코어 포문 적중`

## 3. 카메라 제어 (Camera) 수학 관련 Q&A

### 3.1. 오일러 각(Euler Angles): 벡터 Direction 산출 시 왜 굳이 `cos(yaw)`와 `cos(pitch)`를 연성 곱셈처리 하는가?

FPS 게임 속 플레이어 시점인 카메라의 정면 포인팅 방향 벡터(`direction`)를 산출하려면 모가지 좌우 스윙 각도(yaw)와 상하 끄덕임 각도(pitch)를 융합해야 한다. X축(가로) 방향 성분인 `direction.x` 를 산출하는 수학 방정식은 얄밉게도 이렇게 생겨먹었다.

`direction.x = cos(glm::radians(yaw)) * cos(glm::radians(pitch));`

여기서 좌우각(yaw) 코사인에다가 뜬금없이 상하각(pitch) 코사인 값을 한 번 더 팩터(Factor)로 때려 곱하는 근본적인 이유는, **카메라 고개가 하늘이나 땅으로 수직으로 꺾일수록, 좌우 수평(X, Z축) 방향 평면 위로 투영되는 길다란 시야 벡터의 그림자 길이가 찌그러지듯 짧아지는 물리적 구면 좌표계 현상을 완벽히 보정해 주기 위함이다.**

*   `**cos(yaw)**`: 만약 모가지를 위아래로 까딱하지 않고 지면과 수평(Pitch=0)으로만 부엉이처럼 돌린다면, X축과 Z축이 맞물리는 바닥 수평 평면 좌표계 상에서 카메라 렌즈 방향의 가로 성분 크기를 결정짓는 절대자다.
*   `**cos(pitch)**`: 여기서 고개를 상하로 까딱이는 `pitch` 변수가 개입하여 모가지 각도에 따른 수평 파워 상실 비율을 연대 책임을 지고 깎어 내리는(Discount) 보정 가중치다.
    *   **정면 주시 (pitch = 0°)**: `cos(0) = 1` 이다. 상하 왜곡이 일절 없으니 가중치가 1. 즉 `cos(yaw)` 파워가 100% 온전하게 `direction.x` 성분으로 투입 보존된다.
    *   **수직 승천 뷰 (pitch = 90° 상승)**: `cos(90) = 0` 이다. 카메라가 바로 모가지를 꺾어 머리 꼭대기 천장 정중앙을 직각으로 치켜보고 있다면? 당신의 로컬 축 기준 앞으로 나아가는 수평(X,Z) 벡터 이동 컴포넌트는 완전 제로(0) 소멸 증발해 버려야 정상이다. 보정 가중치가 0이 곱해지면서 `direction.x` 는 깔끔하게 0으로 차단된다. 참으로 아릅다운 수학이다.

결론적으로, **`cos(pitch)`를 연달아 곱해 치는 행위는 렌즈 고개가 위아래 수직으로 들리거나 숙여질 때마다 수평선상으로 뻗어 나가는 좌표 이동 게이지를 점진적으로 압축 상쇄시켜, 시야 방향 벡터가 구(Sphere) 표면 외곽선을 기형 없이 매끄럽게 활공하도록 잡아주는 삼각함수의 절묘한 밸런싱 테크닉이다.**

### 3.2. 마우스 XY 오프셋 마이너스/플러스 계산 부호 규칙이 서로 엇갈리는 아이러니의 원인

카메라 렌즈 회전 가속도를 연산하기 위해 모니터 커서의 이전 틱 프레임 위치(`lastX, lastY`)와 현재 갓 들어온 프레임 틱 위치(`xpos, ypos`) 간의 이동 변위량(오프셋) 델타 값을 빼기 연산으로 산출한다. 그런데 코드를 자세히 보면 X와 Y의 차감 피연산자 순서가 이상야릇하게 반대로 크로스 되어있다.

```cpp
float xoffset = xpos - lastX;
float yoffset = lastY - ypos; // 엇? 왜 ypos - lastY가 아니지?
```
{: file="main.cpp" }

이 모순의 원흉은 근본적으로 **GLFW OS 라이브러리와 인간의 일반적인 컴퓨터 시각 좌표계 패러다임이 Y축 상하 반전 역상을 이루고 있기 때문**이다.

1.  **GLFW 스크린 좌표계 (데카르트 근본 수학 기준)**: 컴퓨터 그래픽스 라이브러리인 GLFW는 교과서 수학 함수 그래프처럼 좌측 최하단 모서리 구석탱이를 절대 원점 `(0, 0)` 로 규정한다. X축은 우측으로, **Y축은 하늘 우주로 솟구칠수록 양수(+)로 우상향 증가**하는 기틀을 닦아놓았다.
2.  **모니터/웹 디스플레이 DOM 윈도우 OS 좌표계 (인간 시각 기준)**: 반면 모니터 화면 좌표계는 무조건 좌상단 꼭짓점을 시그니처 픽셀 `(0, 0)` 원점으로 박아버린다. 텍스트 읽는 방향을 쫓아가듯, Y축은 **땅바닥 아래로 수직 하강할수록 값이 양수(+)로 더 가라앉아 증가**한다. 
3.  **X 오프셋 (평화)**: 마우스를 오른쪽으로 밀면 모니터나 GLFW나 만장일치로 `xpos` 값이 절대적으로 커진다. 따라서 최신 위치 `xpos` 에서 과거 위치 `lastX` 를 차감(`xpos - lastX`)하면 아름다운 양수 오프셋 델타값이 튀어나온다.
4.  **Y 오프셋 (패러독스 반골 연산)**: 여기서 골때린다. 만약 마우스를 북쪽 하늘로 확 튕겨 올리면(인게임 캐릭터 고개를 하늘로 치켜들 의도로), 모니터 OS 기반 좌표로는 `ypos` 픽셀값이 오히려 100에서 20으로 확 줄어들어 감소해버린다. 
    * 만일 멍청하게 앞선 X축 공식대로 `ypos - lastY` (즉 $20 - 100$)를 때려버리면 오프셋 결과 변위가 $-80$ 등 음수(-) 역배열이 뜨게 되고, 이를 그대로 Pitch각에 더해버리면 인게임 카메라 시야 고개가 땅바닥 낭떠러지로 수직 곤두박질치는 대역전 멀미 쇼가 펼쳐진다.
    * 사용자가 진짜 의도한 "마우스 윗방향 긋기 = 화면 시야 하늘 상승(양수 Pitch 조작)" 이라는 조작 감각 통일 매커니즘을 맞추어 주기 위해, 개발자는 고의적으로 Y 계산식의 피연산자 자리를 `lastY - ypos` ($100 - 20 = 80$) 로 비틀어 꼬아 도출된 오프셋 부호 마이너스/플러스 여부를 억지로 180도 뒤집어 반전 파싱하는 매직을 부리는 것이다.
